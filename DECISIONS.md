# Принятые технические решения (DECISIONS.md)

В ходе выполнения задания были приняты следующие архитектурные и технические решения:

1. **Технологический стек (PHP 8.2+ / Laravel 11 / PostgreSQL):**
   Laravel был выбран как наиболее эффективный фреймворк для быстрой разработки бизнес-логики с использованием встроенной ORM Eloquent, которая предоставляет надежные инструменты для работы с транзакциями.

2. **Контейнеризация (Docker Compose):**
   Приложение развернуто через Docker для обеспечения идентичности окружения на Mac (среда разработки) и Linux (среда проверки). Использован образ `php-fpm-alpine` для минимизации размера контейнера.

3. **Защита от Race Condition (Pessimistic Locking):**
   Для реализации безопасного действия «Взять в работу» использована пессимистическая блокировка `lockForUpdate()` внутри БД-транзакции. 
   - **Почему:** Это гарантирует, что в момент проверки статуса и его обновления строка заблокирована для других процессов. Альтернатива (Optimistic Locking) была отклонена из-за необходимости добавления полей версионности и сложности обработки повторных попыток.

4. **Обработка ошибок параллелизма:**
   В случае, если заявка уже была взята другим мастером, сервер возвращает статус `409 Conflict`. Это семантически корректный HTTP-ответ для ситуации, когда запрос не может быть выполнен из-за текущего состояния ресурса.

5. **Модель данных и роли:**
   Для упрощения авторизации (согласно ТЗ) использовано поле `role` в таблице `users`. Это позволило избежать избыточности сложных систем контроля доступа (RBAC), сохранив при этом четкое разделение логики Диспетчера и Мастера.

6. **Система статусов (State Management):**
   Жизненный цикл заявки реализован через `enum` в БД, что гарантирует целостность данных и предотвращает запись некорректных статусов.

7. **Тестирование (Feature Tests):**
   Написаны автоматические тесты для проверки ключевого функционала: создание заявки и смена статусов, что обеспечивает стабильность базовых функций.
